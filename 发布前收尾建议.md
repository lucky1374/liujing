# 发布前收尾建议

## 1. 当前检查结果

已确认通过：

- 后端 `ai-test-platform` 可成功构建
- 前端 `ai-test-platform-web` 可成功构建
- 真实项目、真实环境、真实 Bearer Token 已验证
- `lite` 与 `python` 两种执行模式均已验证

## 2. 当前仍建议收口的事项

### 2.1 本地启动脚本路径写死

当前文件：

- `/Users/lj/liujing/start-local.sh`
- `/Users/lj/liujing/stop-local.sh`

问题：

- 脚本中使用了固定绝对路径 `/Users/lj/liujing`
- 适合当前机器，不适合直接给其他人复用

建议：

- 后续改成基于脚本所在目录自动推导根路径

### 2.2 前端打包体积偏大

当前构建存在告警：

- `user-*.js` 超过 500kB

影响：

- 不阻塞当前内部演示
- 但后续建议拆分 chunk，优化首屏加载

建议：

- 按页面做更明确的异步加载
- 排查 Pinia / 业务聚合模块是否过于集中

### 2.3 `.env.example` 仍可继续补充

当前建议补充但不阻塞：

- `PYTHON_BIN`
- `SEED_ADMIN_USERNAME`
- `SEED_ADMIN_PASSWORD`
- `SEED_ADMIN_EMAIL`

### 2.4 Docker 本地方案未实际验证

现状：

- 已补 `docker-compose.local.yml`
- 当前机器没有 `docker` 命令，未做实际启动验证

建议：

- 后续在支持 Docker 的环境中验证一次

### 2.5 初始迁移只适用于新库

现状：

- 已生成 `InitSchema` 初始迁移
- 当前开发库可能仍依赖 `synchronize`

建议：

- 当前开发库继续使用
- 新环境/新库从 `InitSchema` 起步
- 后续所有结构变更走增量迁移

## 3. 当前版本可作为什么使用

当前版本已经适合：

- 内部演示版
- 真实项目验证版
- 小范围试运行版
- 下一阶段研发基线版本

当前版本暂不建议直接定义为：

- 大规模多团队共用生产版
- 高并发正式执行平台

## 4. 最推荐的下一步顺序

1. 把本地启动脚本改成通用相对路径版本
2. 补 `.env.example` 的 Runner/Seed 配置项
3. 继续做增量迁移规范
4. 在支持 Docker 的机器上补一次 compose 验证

## 5. 结论

当前平台已经达到“可展示、可联调、可继续扩展”的阶段。发布前最大的风险已不在功能链路，而在通用化、部署一致性和后续工程收口。
